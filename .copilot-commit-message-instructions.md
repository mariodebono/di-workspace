# Conventional Commit Instructions

Use the Conventional Commits 1.0.0 specification for every generated commit message.

## Format

Write the commit message as:

```text
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

## Rules

- Start with a type such as `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, or `revert`.
- Use `feat` for a new feature and `fix` for a bug fix.
- Optionally include a scope in parentheses when it adds useful context, such as `feat(parser): ...`.
- Follow the type and optional scope with `: ` and a short, imperative description.
- Keep the description concise and specific.
- Do not end the header with a period.
- Use a body only when extra context is helpful. Leave one blank line between the header and the body.
- Use footers only when needed. Follow git trailer style, one blank line after the body.
- Mark breaking changes with `!` before the colon, or with a footer that starts with `BREAKING CHANGE: `.
- If `!` is used in the header, the breaking change details should be described in the header description or in the body.
- Prefer lower-case types for consistency.
- If multiple footers are needed, separate them with blank lines.

## SemVer mapping

- `fix` maps to a patch release.
- `feat` maps to a minor release.
- Any commit with a breaking change maps to a major release.

## Examples

```text
feat: add commit message prompt instructions
```

```text
fix(parser): handle empty commit scopes
```

```text
feat!: remove support for legacy commit format

BREAKING CHANGE: commit messages must now follow Conventional Commits 1.0.0
```

## Guidance

- Prefer the smallest correct commit type.
- If a change touches multiple areas, choose the most significant user-facing type.
- If the change is purely documentation, use `docs`.
- If the change is housekeeping or tooling, use `chore` unless another type is more accurate.
